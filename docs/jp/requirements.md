# PartWire 要件定義書

## 1. 文書の目的

本書は、内製設備の購入部品を対象に、見積もり依頼管理、注文管理、納品管理、請求書管理までを一貫して行うアプリケーション `PartWire` の要件を整理することを目的とする。

本アプリは、少人数運用でも案件単位と工事番号単位の両方で進捗と金額を把握できること、ならびに納品確認業務を素早く正確に行えることを重視する。

## 2. 背景と課題

- 購入部品に関する情報が、見積もり、発注、納品、請求の各段階で分散しやすい
- 工事番号単位での集計や案件全体の追跡がしづらい
- 部品価格の履歴が蓄積されず、次回見積もり時の参考情報が残りにくい
- 納品確認の業務負荷が高く、現場で素早く確認しづらい
- 各種マスターや既存データの初期取り込みを柔軟に行いたい

## 3. システム化の目的

- 案件の起票から後工程への書類回付までを一つのアプリで管理する
- 工事番号ごと、案件ごと、部品ごとに必要な情報を追跡できるようにする
- 見積もり価格と実績価格を蓄積し、将来の調達判断に活用できるようにする
- 納品確認を短時間で実施できる操作性を提供する
- CSV による一括登録・更新で、マスター管理とデータ移行を容易にする

## 4. 想定利用者

- 依頼者
- 手配対応者
- 管理者

補足:
- 実運用人数は 5 名程度を想定する
- 1 人が複数の役割を兼務する可能性がある

## 5. 業務対象範囲

本システムの対象業務は以下とする。

- 上位の工事番号登録
- 案件登録と案件番号管理
- サプライヤーへの見積もり依頼管理
- 見積もり結果の登録
- 購入承認依頼
- サプライヤーへの発注
- 納期登録
- 納品書登録
- 納品確認
- 請求書登録
- 納品書、請求書の後工程回付登録
- 部品マスタ管理
- 製作部品管理
- 価格履歴管理
- 各種マスタおよびデータの CSV 取込

## 6. 業務フロー

業務フローの基本形は以下とする。

1. 依頼者が案件登録を行う
2. 担当者がサプライヤーへ見積もり依頼を行う
3. 担当者が見積もり内容を登録する
4. 担当者が購入承認依頼を行う
5. 担当者がサプライヤーへ発注を行う
6. 担当者が納期を登録する
7. 担当者が納品書を登録する
8. 担当者が納品確認を行う
9. 担当者が請求書を登録する
10. 担当者が請求書、納品書を後工程に回したことを登録する

補足:
- 請求書登録は一部サプライヤーのみ対象とする
- 実際の運用では一案件に複数明細、複数部品、複数サプライヤーが存在し得る

## 7. 機能要件

### 7.1 案件管理

- 上位の工事番号を登録できること
- 独自の案件番号を自動採番で登録できること
- 案件に対して依頼者と手配対応者を設定できること
- 1 つの案件は必ず 1 つの工事番号に紐づくこと
- 案件の状態を業務フローに沿って管理できること
- 案件に紐づく見積、発注、納品、請求の履歴を参照できること
- 案件登録時点で部品明細まで登録できること
- 案件の部品明細は見積取得依頼の元データとして扱えること
- 案件途中で部品明細を追加できること

### 7.2 工事番号管理

- 工事番号の登録、更新、無効化ができること
- 工事番号単位で案件件数、見積金額、発注金額、納品金額、請求金額を集計できること
- 集計期間や状態による絞り込みができること

### 7.3 部品マスタ管理

- 購入対象部品を部品マスタとして登録できること
- 最低限、メーカー名と型式を登録できること
- 必要に応じて部品名、単位、備考、標準サプライヤーなどを保持できること
- 重複部品を識別しやすいこと

### 7.4 製作部品管理

- 製作部品を購入部品とは別区分で管理できること
- 製作部品について、図番、品名、材質、形状、加工法、サイズ、表面処理を登録できること
- 図番と品名は必須入力とすること
- 製作部品の管理項目は将来的に増やせること
- 材質、形状、加工法、サイズ、表面処理などは未入力でも登録できること
- 製作部品の属性を加工メーカーの絞り込み条件として利用できること
- 加工メーカーの絞り込みは補助的な用途とし、属性未入力でも運用できること
- アプリを利用する会社ごとに、製作部品の管理項目を拡張しやすいこと
- サイズは自由入力値ではなく、主にサイズ帯を表すマスタとして管理できること
- 加工法は放電、旋盤、フライス盤、マシニング、6軸、3Dプリンタ、研削程度の粒度で管理できること
- 表面処理はアルマイト、無電解ニッケル等のめっきを中心に管理できること

### 7.5 取引先管理

- サプライヤーと加工メーカーを共通の取引先マスタで管理できること
- 1 つの取引先に対して、サプライヤー、加工メーカーの両方の役割を設定できること
- 購入部品の見積先候補には、サプライヤー役割を持つ取引先を優先表示できること
- 製作部品の見積先候補には、加工メーカー役割を持つ取引先を優先表示できること
- 製作部品では、部品属性に応じて加工メーカー候補を絞り込みできること
- 役割や属性条件に合致しない取引先も、必要に応じて手動選択できる余地を持つこと

### 7.6 価格履歴管理

- 部品ごとに価格履歴を登録できること
- 価格履歴に、価格、通貨、適用日、サプライヤー、見積番号などを保持できること
- 見積登録または発注登録から価格履歴を自動または半自動で登録できること
- 最新価格と過去価格を参照できること

### 7.7 見積もり依頼・見積管理

- サプライヤーへの見積依頼日、依頼先、依頼内容を登録できること
- 見積回答日、見積金額、見積有効期限、見積番号を登録できること
- サプライヤー側の見積番号を管理できること
- 複数サプライヤーからの見積比較ができること
- 見積未回答案件を一覧で把握できること
- 1 つの部品明細に対して相見積もりを取得できること
- 採用しなかった見積書も履歴として保持できること
- 採用しなかった見積価格も価格履歴として参照できること
- 見積内容変更時は見積書を版管理できること
- 見積書の各版の履歴を参照できること
- サプライヤー側の見積番号は改版時にも同一のまま継続できること

### 7.8 購入承認依頼

- 見積書ごとに購入承認依頼を起票できること
- 承認依頼日、承認者、承認状態、承認コメントを登録できること
- 承認待ち、承認済み、差戻しを識別できること
- 案件途中で追加取得した見積書のみを個別に承認依頼できること
- 見積書が改版された場合は再承認を必須とすること
- 承認経路は部署と金額ランクに応じて決定できること
- 承認者に承認者ランクを設定できること
- 見積金額に応じて必要な承認者ランクを判定できること
- 案件ごとに適用部署を持ち、承認者選定条件に利用できること
- 承認ルートを手動補正できる余地を持つこと

### 7.9 発注管理

- 承認済み見積書に対して発注登録ができること
- 発注日、発注先、発注番号、発注金額、数量、納期を登録できること
- 分納や追加発注が発生した場合に履歴として追跡できること
- 発注は見積書単位で実施できること
- 発注内容が変更になった場合は、見積書更新または再見積で対応できること
- 改版前の見積書に対して発注済みの場合、発注キャンセルを記録できること
- 見積書改版時は旧版見積書をキャンセル状態として扱えること

### 7.10 納品・納品確認管理

- 納期を登録、更新できること
- 納品書情報を登録できること
- 納品確認を容易に行える画面を提供すること
- 未納、納品済み、部分納品、確認済みなどの状態を判別できること
- 納品確認日、確認者、数量差異、不備内容を記録できること
- 分納を管理できること
- 納品確認は部品ごとの数量確認を必須とすること
- 納品確認は納品書ごとに 1 件ずつ実施しやすいこと
- 納品書登録は手入力を基本とすること
- 納品書登録時に対象の見積書明細を呼び出して入力補助できること
- 見積書明細ごとの残納数量を参照しながら納品書登録できること

### 7.11 請求書管理

- サプライヤーごとに請求書登録要否を設定できること
- サプライヤーまたは加工メーカーごとに、請求書が自社側へ回ってくるかどうかを設定できること
- 請求書番号、請求日、請求金額、支払対象月などを登録できること
- 納品情報との対応関係を確認できること
- 請求書は原則として見積書単位で管理できること
- 月またぎ分納時には見積書に対して請求を分割登録できること
- 見積書の完了判定は明細レベルの納品、請求状況を積み上げて判定できること
- 請求書登録時に対象の納品書または納品明細を呼び出して入力補助できること
- 納品明細ごとの未請求数量や未請求金額を参照しながら請求書登録できること

### 7.12 後工程回付管理

- 納品書、請求書を後工程へ回付したことを登録できること
- 標準では見積書は後工程回付対象としないこと
- 会社設定により見積書も後工程回付対象に変更できること
- 回付日、回付先、回付者、備考を記録できること
- 未回付案件を一覧で確認できること

### 7.13 CSV 取込・出力

- 各マスターを CSV で一括登録、更新できること
- 購入部品マスタ、製作部品マスタを CSV で一括登録、更新できること
- 案件、見積、発注、納品、請求などの業務データも CSV 取込できること
- 取込前に項目対応とエラーチェックができること
- エラー行を利用者が把握できること
- 集計や外部提出向けに CSV 出力できること

### 7.14 検索・一覧・集計

- 案件番号、工事番号、部品、サプライヤー、依頼者、担当者、状態で検索できること
- 工事番号ごとに金額と件数を集計できること
- 未処理案件、納品待ち、請求待ちなどの一覧を表示できること

## 8. 画面要件の方向性

- トップ画面で、担当者ごとの未処理件数と注意案件を把握できること
- 納品確認画面は、少ない操作で対象案件を絞り込み、確認登録できること
- 納品書登録画面では、対象見積書の明細を呼び出して納品数量を入力しやすいこと
- 請求書登録画面では、対象納品書の明細を呼び出して請求対象を入力しやすいこと
- 案件詳細画面で、見積から回付までの進捗を時系列で確認できること
- CSV 取込画面では、テンプレート取得、プレビュー、エラー確認ができること

## 9. 非機能要件

### 9.1 運用性

- 5 名程度で無理なく運用できること
- 特定個人しか使いこなせない複雑な操作にしないこと
- メンテナンスしやすいマスター構成とすること

### 9.2 性能

- 日常利用でストレスのない検索速度を確保すること
- 納品確認画面は特に応答性を重視すること
- CSV 取込時は数百件から数千件規模を現実的な時間で処理できること

### 9.3 技術構成

- 確認用のローカルスタンドアロン版として SQLite に対応できること
- 複数人利用向け構成として SQL Server に対応できること
- DB 切替により業務機能差分を極力発生させないこと
- SQLite から SQL Server への移行を可能とすること
- 認証方式としてアプリ独自認証と AD 認証の両方に対応できること
- 認証方式は設定により切り替えできること
- Active Directory 認証ではログオン連携まで対応できること
- アプリ配布はインストーラー方式とすること

### 9.4 監査性

- 誰がいつ登録、更新、確認したかを追跡できること
- 金額や状態変更の履歴を保持できること
- 中間テーブルを除く主要テーブルの CRUD 操作を監査できること

### 9.5 セキュリティ

- 利用者を識別できること
- 役割に応じて操作範囲を制御できること
- 権限制御を細かく設定できること
- 削除よりも無効化や履歴保持を優先すること
- 独自に管理する利用者情報と権限情報を保持できること
- ログイン認証方式として Active Directory 認証を利用できること
- ログイン認証方式としてアプリ独自のユーザーID、パスワード認証を利用できること
- 認証方式はシステム設定により切り替えできること
- 認証方式を切り替えても、業務上の利用者、権限、承認者情報を継続利用できること
- Active Directory のグループ情報を参照して権限制御や承認者候補抽出に利用できること

### 9.6 添付ファイル管理

- 見積書、納品書、請求書などの関連ファイルを任意で紐づけできること
- 添付ファイルは必須ではないこと
- DB にはファイル本体を保存せず、保存先リンクまたはパスのみを保持すること
- 添付リンク切れを検知または利用者が識別しやすいこと
- 添付ファイル保存先は設定により切り替えできること
- 標準の添付ファイル保存先は共有フォルダとすること

### 9.7 可用性、運用

- DB 接続先や添付保存先などの接続先設定を変更できること
- 接続障害時は通常操作を停止してもよいこと
- バックアップ運用は基本的に運用者に委ねること
- ただしバックアップ対象と推奨手順を明示できること
- 初期データ投入用にサンプル読み込み機能を提供できること

## 10. 現時点で不足していそうな要件

以下は実装前に詰めておきたい論点である。

### 10.1 案件と工事番号の関係

- 工事番号は最上位の集計単位とし、1 工事番号に複数案件が属する
- 1 案件は必ず 1 つの工事番号に属する

### 10.2 案件番号の採番ルール

- 自動採番とする
- 案件番号はサロゲートキー相当とし、年度や部署などの業務的意味を持たせない
- 利用者が入力する可能性を考慮し、GUID のような長い識別子は採用しない
- 重複しない連番または短い一意識別子とする
- 将来の仕様変更時にも採番ルール変更の影響を局所化できること

### 10.3 明細粒度

- 案件は複数の部品明細を持つ
- 見積は見積書単位で管理し、進捗と完了判定は明細単位で追跡する
- 発注は見積書単位で行う
- 納品確認は部品明細単位で行う
- 請求は見積書単位を基本とするが、分割請求を許容する

### 10.4 承認フロー

- 承認は見積書単位で行う
- 案件途中の追加見積は、その追加分のみ個別承認する
- 承認者は部署と金額ランクで変動する
- 承認者に承認者ランクを持たせる
- 1 段階承認か複数段階承認か
- 金額閾値による承認ルート変更がある
- 承認ルートを個別案件で上書きできるか

### 10.5 納品確認の運用

- 現場でバーコードや QR コードを使うか
- 納品書ごとに 1 件ずつ確認する運用を前提とする
- PC 前提か、タブレット利用も想定するか

### 10.6 請求書管理の対象範囲

- 請求書登録が必要なサプライヤーの判定ルール
- 請求書 1 件に対して複数納品を紐づける必要があるか
- 見積書単位の請求完了判定をどの条件で行うか
- 請求書受領要否を取引先マスタでどう管理するか

### 10.7 書類ファイル管理

- 見積書、納品書、請求書の PDF や画像添付が必要か
- 添付ファイルの保存場所と容量上限をどうするか
- リンク先をローカル共有フォルダ、ファイルサーバー、SharePoint 等のどれにするか

### 10.8 通知要件

- 見積回答遅延、納期超過、未回付などの通知が必要か
- メール通知か、アプリ内通知か

### 10.9 権限要件

- 依頼者、担当者、管理者で閲覧範囲や編集範囲を分けるか
- 金額閲覧可否や承認可否を役割で制御するか
- 独自ユーザーと AD グループのどちらを権限判定の主とするか
- AD グループ参照失敗時の代替運用をどうするか
- 認証方式切り替え時の利用者ひも付けをどう維持するか

### 10.10 CSV 取込ルール

- 新規追加のみか、更新・上書きも許容するか
- キー項目を何にするか
- 取込エラー時のロールバック範囲をどうするか

### 10.11 製作部品属性の拡張ルール

- 会社ごとの追加属性をマスタ設定で増やせるようにするか
- 固定カラム追加で対応するか、汎用属性方式で対応するか
- 加工メーカーのフィルター条件に使う属性をどこまで標準搭載するか

## 11. 次に決めたい事項

優先度の高い確認項目は以下とする。

1. 承認フローの具体像
2. 納品確認画面の運用イメージ
3. 請求書の管理範囲
4. 添付ファイル有無
5. 案件番号の具体的な採番桁数と表示形式
6. 発注キャンセルと改版時の運用詳細
