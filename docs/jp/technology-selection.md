# PartWire 技術選定メモ

## 0. 目的

本書は、`PartWire` の技術選定を進めるための判断メモである。

- コンセプトに合う技術を優先する
- OSS 公開前提で、理解しやすく参加しやすい構成を目指す
- v1 で決める項目と、後続で詰める項目を分けて整理する

本書は確定仕様書ではなく、設計判断を前に進めるための整理資料として扱う。

## 1. 技術選定の前提

PartWire の前提は次のとおり。

- クラウド契約や導入承認が通りにくい部署でも使えること
- 単一 PC で運用開始できること
- 環境がある組織では、サーバー上の DB を利用できること
- Windows PC を前提に、日常業務を素早く回せること
- 少人数運用でも無理がないこと
- OSS として公開し、将来の改善余地を残すこと

この前提から、技術選定の判断軸は次のように置く。

- Windows 業務アプリとして操作しやすいこと
- ローカル運用から段階的に拡張できること
- 実装や保守が複雑になりすぎないこと
- OSS として構成が理解しやすいこと
- DB 差し替えや将来拡張に備えて責務分離できること

## 2. 現時点の大枠判断

現時点では、以下の方向で進めるのが妥当と考える。

### 2.1 アプリ方式

- 採用方針: Windows デスクトップアプリ
- UI 方式: WPF

理由:

- 本プロダクトは Windows PC 前提の業務アプリである
- 単一 PC 運用の開始がしやすい
- キーボード主体の一覧入力や確認画面を作りやすい
- サーバー常設を前提にしない構成と相性がよい

補足:

- OSS としては Web より参加者層が狭くなりやすい
- そのため、UI と業務ロジックを分離し、将来の別 UI 実装余地は残す

### 2.2 ランタイム

- 採用方針: .NET 10

理由:

- チーム内に実績があり、開発判断と保守判断をそろえやすい
- 新しい .NET の機能や改善を前提に実装しやすい
- WPF、SQLite、SQL Server、主要ライブラリとの相性がよい

補足:

- 現在のプロジェクトは `net10.0-windows` であり、方針とも一致している
- OSS 公開時は対応 SDK バージョンを README と開発手順に明記する

### 2.3 アーキテクチャ

- 採用方針: UI と業務ロジックを分離したレイヤード構成

初期案:

- `PartWire.Desktop`: WPF 画面、画面遷移、View/ViewModel
- `PartWire.Application`: ユースケース、サービス、入力検証
- `PartWire.Domain`: エンティティ、値オブジェクト、業務ルール
- `PartWire.Infrastructure`: DB、ファイル、認証、通知など外部接続

理由:

- WPF 固有の都合を業務ルールに持ち込まないため
- SQLite と SQL Server の差分吸収位置を明確にするため
- OSS として読んだときに責務が分かりやすいため
- 将来 Web UI や CLI ツールを追加しやすくするため

### 2.4 画面パターン

- 採用方針: MVVM

### 2.5 WPF フレームワーク方針

- 第一候補: `CommunityToolkit.Mvvm` + `Microsoft.Extensions.Hosting`
- 補助候補: `Microsoft.Extensions.DependencyInjection`

理由:

- WPF で必要な MVVM 支援を軽量に導入できる
- `ObservableObject`、`RelayCommand`、`AsyncRelayCommand` など基本機能がそろっている
- .NET 標準寄りの DI、設定、ログ構成と合わせやすい
- OSS として読んだときに、独自ルールより構成意図が伝わりやすい
- Prism のような大型フレームワークより学習コストと依存を抑えやすい

Prism を第一候補にしない理由:

- 現時点ではモジュール分割や複雑な画面遷移基盤を先に必要としていない
- 小さく始めるにはやや重い
- PartWire はまず業務ロジックとデータ設計を固める段階であり、UI 基盤を厚くしすぎない方がよい

現時点の提案:

- ViewModel: `CommunityToolkit.Mvvm`
- DI / 起動基盤: `Generic Host`
- ログ: `Microsoft.Extensions.Logging` + `Serilog`
- 設定: `Microsoft.Extensions.Configuration`

この組み合わせで、WPF の素直さを保ちながら、将来の拡張にも耐えやすい構成にできる。

理由:

- WPF では最も理解しやすく、保守しやすい構成の一つである
- OSS 参加者にも意図が伝わりやすい
- コードビハインドへの業務ロジック流出を防ぎやすい

補足:

- 画面数が増えるため、早い段階で ViewModel と画面状態管理の方針を固めた方がよい

## 3. DB とデータアクセス

### 3.1 DB 方針

- v1 の基本構成: SQLite
- 推奨運用構成: SQL Server

整理:

- 単一 PC で開始する場合は SQLite を使えるようにする
- 組織側に環境がある場合は SQL Server を推奨する
- 業務機能は DB 製品差分をできるだけ受けないようにする

理由:

- コンセプト上、最初からサーバー前提にはしない
- 一方で、複数人利用や保守性を考えるとサーバー DB の価値は高い
- 「まず動かす」と「ちゃんと運用する」を両立しやすい

### 3.2 ORM / データアクセス

- 採用方針: Entity Framework Core

理由:

- SQLite と SQL Server の両方を公式プロバイダーで扱いやすい
- マイグレーション管理と初期構築の整備がしやすい
- OSS として学習コストが比較的低い
- 楽観ロック、監査、トランザクションなどの実装基盤を作りやすい

補足:

- 一覧検索や集計など、性能要件が厳しい箇所は SQL を個別最適化してよい
- その場合も、全面的に ORM を捨てるのではなく、読み取り専用クエリを分離する

推奨方針:

- 更新系は EF Core を主軸にする
- 重い一覧や集計は必要に応じて専用クエリを追加する

### 3.3 マイグレーション

- 推奨方針: EF Core Migration を基本とする
- SQLite から SQL Server への移行は別途ツール化する

理由:

- 新規環境の構築手順を簡単にできる
- OSS 利用者が初期セットアップしやすい
- 既存方針の「専用移行ツール」とも整合する

## 4. 認証と権限

### 4.1 v1 の優先順位

- v1 先行: ローカル認証
- v1 以降または段階導入: Active Directory 認証

理由:

- OSS の初期利用者がすぐ試せるのはローカル認証である
- AD 連携は有用だが、検証条件や導入条件が限定される
- まずはアプリ単体で動くことを優先した方が価値を出しやすい

補足:

- データモデル上は最初から UPN を保持できるようにする
- 認証方式切り替えを見据え、認証ロジックはインターフェースで分離する

### 4.2 権限制御

- 推奨方針: ロールベースを中心に実装する
- 追加条件: 部署、承認者ランク、取引先関連制御をアプリ層で補完する

理由:

- 要件上の説明と一致していて分かりやすい
- 初期実装として過不足が少ない

## 5. 添付、通知、監査

### 5.1 添付ファイル

- v1 方針: ローカルパスまたは共有フォルダパスを保持する
- 後続候補: SharePoint 対応

理由:

- 単一 PC 運用とサーバー DB 運用の両方に乗せやすい
- 実装と運用の複雑さを抑えやすい

### 5.2 通知

- v1 方針: まずはアプリ内通知を優先
- 後続候補: メール通知

理由:

- ローカル運用でも成立しやすい
- SMTP や外部通知基盤の前提を減らせる

### 5.3 監査

- v1 方針: DB ベースで主要テーブルを監査する

理由:

- 業務要件に合致する
- DB 差し替え後も追従しやすい
- アプリログと役割を分離しやすい

## 6. 配布と運用

### 6.1 配布方式

- 第一候補: インストーラー配布
- 検討候補: MSIX または WiX

整理:

- OSS 配布では zip 展開よりもインストーラーの方が導入しやすい
- ただし証明書、更新、企業内配布の扱いまで含めて比較が必要

現時点の考え方:

- まずは開発と検証を優先する
- 配布方式は v1 リリース前に比較決定する

### 6.2 ログ

- 推奨候補: `Microsoft.Extensions.Logging` + `Serilog`

理由:

- .NET で一般的な構成であり、OSS として理解しやすい
- ファイル出力や障害調査に使いやすい

## 7. v1 推奨スタック

現時点での v1 推奨スタックは次のとおり。

- UI: WPF
- ランタイム: .NET 10
- 画面設計: MVVM
- WPF フレームワーク: CommunityToolkit.Mvvm + Generic Host
- DB: SQLite を基本対応、SQL Server を推奨対応
- データアクセス: EF Core
- ログ: Microsoft.Extensions.Logging + Serilog
- CSV: CsvHelper
- 配布: インストーラー方式

## 8. 先に決めたい項目

次の論点は、実装に入る前に優先して決めたい。

1. WPF フレームワークを CommunityToolkit.Mvvm + Generic Host で進めるか
2. v1 の認証をローカル認証先行とするか
3. v1 の通知をアプリ内通知優先にするか
4. インストーラー候補を MSIX と WiX のどちらから検証するか

## 9. 現時点の提案

現時点では、次の組み合わせで前進するのが最も無理が少ない。

- WPF
- .NET 10
- レイヤード構成
- MVVM + CommunityToolkit.Mvvm + Generic Host
- EF Core
- SQLite 先行、SQL Server 対応可能な構成
- ローカル認証先行
- 添付はローカル/共有フォルダパス方式
- 通知はまずアプリ内通知

この構成なら、単一 PC でも始められ、環境がある組織では SQL Server へ広げやすい。また、OSS としても「Windows 向けのローカルファースト業務アプリ」という説明がしやすい。
