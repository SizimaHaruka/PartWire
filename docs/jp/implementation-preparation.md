# PartWire 実装準備メモ

## 0. 目的

本書は、実装開始前にそろえておきたい成果物と判断項目を整理するためのメモである。

現在の前提は次のとおり。

- UI は WPF
- ランタイムは .NET 10
- アーキテクチャはレイヤード構成
- データアクセスは EF Core
- 単一 PC 運用を可能にしつつ、環境がある場合は SQL Server を推奨する

## 1. 実装開始前に作る成果物

実装開始前に、少なくとも以下を用意してから着手する。

1. DDL
2. クラス設計
3. 画面遷移案

加えて、手戻り防止のため次も先に用意した方がよい。

4. ソリューション構成案
5. DbContext / Repository / Query の責務整理
6. 命名規約と配置規約
7. 初期マイグレーション方針
8. 認証・権限制御の最小実装方針
9. 監査ログと共通カラム方針
10. サンプルデータ方針

## 2. DDL で決めること

DDL は単なるテーブル定義ではなく、業務ルールの実装境界を決める資料として扱う。

最低限、次を含める。

- テーブル一覧
- カラム一覧
- PK / FK
- 一意制約
- インデックス
- NULL 許可方針
- 監査用共通カラム
- 楽観ロック用カラム
- 論理削除または無効化方針

特に先に決めたい点:

- 案件、見積、発注、納品、請求の主キー命名
- 更新日時を楽観ロックにどう使うか
- SQLite と SQL Server で型差分が出やすい項目の扱い
- 金額、数量、日付、真偽値の型方針
- 添付リンク、通知、監査ログのテーブル粒度

## 3. クラス設計で決めること

クラス設計は、テーブル対応クラスだけでは不足する。少なくとも次の観点を分ける。

- Domain Entity
- Value Object
- Application Service / UseCase
- Infrastructure Entity Mapping
- ViewModel
- 画面 DTO / 検索条件 DTO / 一覧表示 DTO

最初に整理したいもの:

- 主要エンティティの責務
- ステータス遷移ロジックの置き場所
- 集計系クエリの扱い
- 非正規化カウンター再計算の責務
- 監査ログ書き込みの責務

補足:

- EF Core の Entity をそのまま Domain Model にしない方が整理しやすい
- ただし v1 では過度にクラスを増やしすぎず、責務分離を優先する

## 4. 画面遷移案で決めること

画面遷移案では、見た目より導線を先に固める。

最低限ほしい画面群:

- ログイン
- ダッシュボード
- 案件一覧
- 案件詳細
- 見積登録
- 承認申請 / 承認確認
- 発注登録
- 納品書登録
- 納品確認
- 請求登録
- 回付登録
- マスタ管理
- CSV 取込
- 設定

先に起案したい内容:

- どの画面を主導線にするか
- 案件中心導線と、承認待ち・納品確認待ちの横断導線の関係
- 新規登録、編集、詳細表示を別画面にするか同画面にするか
- ダイアログとフル画面遷移の使い分け
- エラー時、競合時、差戻し時の戻り先

## 5. そのほかに足りないもの

実装前に足りないものとして、次を追加で用意するのがよい。

### 5.1 ソリューション構成図

少なくとも以下のプロジェクト境界を明確にする。

- `PartWire.Desktop`
- `PartWire.Application`
- `PartWire.Domain`
- `PartWire.Infrastructure`

必要に応じて後から追加する候補:

- `PartWire.Contracts`
- `PartWire.Migrations`
- `PartWire.Tests`

### 5.2 コーディング規約

最初に軽くてもよいので決めておきたい。

- フォルダ構成
- 名前空間規約
- ファイル命名
- `async` メソッド命名
- DI 登録規約
- 例外の扱い

### 5.3 DB 初期化方針

次を先に決める。

- アプリ起動時に自動作成するか
- Migration を明示実行するか
- SQLite と SQL Server で初期化手順を分けるか
- サンプルデータ投入をいつ許可するか

### 5.4 テスト方針

最低限、次は実装前に決めたい。

- Domain / Application 層のユニットテストを主軸にする
- UI は重要導線のみ確認対象にする
- DB 差分確認のため、SQLite と SQL Server で最低限の検証観点を持つ

### 5.5 エラーハンドリング方針

WPF では後回しにすると広がりやすいため、最初に決める。

- 画面向けエラー表示
- 想定業務エラーと想定外例外の分離
- 楽観ロック競合時の再読込導線
- ログ出力粒度

## 6. 実装前チェックリスト

実装を始める前に、次がそろっていれば安全に入りやすい。

- DDL 初版
- 主要エンティティのクラス設計初版
- 主要画面の遷移図
- ソリューション構成図
- 命名規約
- DbContext の責務整理
- Migration 方針
- 認証方針の v1 切り分け
- 監査ログの最小方針
- サンプルデータ投入方針

## 7. 現時点の進め方提案

進め方としては次の順がよい。

1. DDL 初版を作る
2. DDL をもとに Domain / Infrastructure のクラス設計を起こす
3. 主要画面の画面遷移案を作る
4. ソリューション構成とプロジェクト境界を確定する
5. EF Core の初期モデルと Migration 方針を固める
6. その後に実装へ入る

この順なら、DB、クラス、UI 導線の依存関係を整理しながら進められる。
