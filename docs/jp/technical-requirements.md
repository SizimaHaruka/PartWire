# PartWire 技術要件書

## 0. 文書運用

- 本書は認証、DB 差分吸収、監査、添付、移行、更新制御などの技術判断を記録する成果文書として扱う
- 未決事項の一次ソースは [design-decisions-checklist.md](/C:/wk/csherp/PartWire/docs/jp/design-decisions-checklist.md) とする
- 業務運用上の論点は原則として本書へ転記せず、技術判断を伴うものだけを反映する
- 本書への反映は、関連する技術領域がまとまって確定した時点で実施する

## 1. 目的

本書は、`PartWire` の実装、運用、配布に関する技術要件を整理することを目的とする。

## 2. 対応構成

### 2.1 アプリ構成

- クライアントアプリは Windows デスクトップアプリとして提供する
- 配布形式はインストーラー方式とする

### 2.2 DB 構成

- 確認用のローカルスタンドアロン版では SQLite を利用できること
- 複数人利用版では SQL Server を利用できること
- DB 製品を切り替えても、業務機能差分を極力発生させないこと
- SQLite から SQL Server への移行を可能とすること

## 3. 認証、認可

### 3.1 認証方式

- ローカル認証と Active Directory 認証の両方に対応すること
- 認証方式は設定で切り替えできること
- Active Directory 認証ではログオン連携まで対応すること
- Active Directory 認証時の利用者ひも付けキーは UPN とすること

### 3.2 権限制御

- 権限制御は細かく設定できること
- 役割、部署、承認者ランク、取引先関連操作などを踏まえて制御できること
- 画面単位、機能単位、操作単位で制御できること
- 閲覧範囲や操作範囲はロール設定で制御できること
- 権限判定は通常時、独自ユーザー情報と AD グループ参照を併用して解決すること
- AD グループ参照失敗時は独自ユーザー情報のみで継続運用できること
- 金額閲覧のみを個別制御する専用権限機能は設けないこと

## 4. データ更新制御

- 複数人利用時の同時更新制御として楽観ロックを実装すること
- 更新競合時は利用者に再読込または再入力を促せること
- 更新競合時は競合ありと表示し、画面更新のうえ再入力を促すこと
- 楽観ロック判定用の更新日時を各主要テーブルに保持できること
- 非正規化カウンターは関連明細の登録、更新、取消と同一トランザクションで再集計更新すること

## 5. 添付ファイル

- 添付ファイル保存先は設定により切り替えできること
- 標準の保存先は共有フォルダとすること
- SharePoint を添付ファイル保存先の対象に含められること
- DB にはファイル本体を保存せず、リンクまたはパスのみを保持すること
- v1 では見積書、納品書、請求書の添付を実装対象とすること
- 添付容量上限を設け、上限超過時は通知のうえ添付追加のみ停止すること
- 添付を表示する画面では、リンク切れを識別できる表示を行うこと
- 保存先切替時も既存リンクは移行せず、そのまま保持すること

## 6. 監査ログ

- 中間テーブルを除く主要テーブルの CRUD 操作を監査対象とする
- 少なくともマスタ、案件、見積、承認、発注、納品、請求、回付、利用者、設定の操作を記録すること
- 監査ログには対象テーブル、対象ID、操作種別、操作日時、操作者、変更前、変更後を保持すること
- 監査ログは変更前後の主要項目を保持する標準粒度とすること
- 監査ログの保持期間は標準 5 年とし、設定変更可能にすること
- 監査ログの閲覧権限はシステム管理者のみに限定すること

補足:
- 中間テーブルをすべて記録対象にすると件数が過大になりやすいため、まずは主要テーブル中心でよい
- ただし承認ルート定義や取引先対応条件のように業務影響が大きい設定系テーブルは監査対象とする

## 7. 障害時運用

- DB 接続先や添付保存先などの接続先設定を変更できること
- 接続障害時は通常の業務操作を継続できなくてもよい
- 障害復旧後に設定変更または再接続で復帰できること

## 8. データ初期投入

- 初期データ投入用にサンプル読み込み機能を提供すること
- サンプル読み込みは CSV 手動投入ではなく、アプリ内ロジックにより実行できること
- マスタ初期値や検証用データを環境構築直後に投入できること
- CSV 取込では `UserKey` を用いた上書き更新および無効化を行えること
- CSV 取込エラー時は全件ロールバックとすること

## 9. バックアップ

- バックアップ運用は基本的に運用者に委ねる
- ただし、バックアップ対象と推奨手順を運用手順書で提示できること
- SQLite 版では DB ファイルと添付保存先の保全対象を明示できること
- SQL Server 版では DB バックアップと添付保存先の保全対象を明示できること
