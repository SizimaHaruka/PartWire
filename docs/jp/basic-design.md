# PartWire 基本設計書

## 0. 文書運用

- 本書は基本設計として、エンティティ、状態遷移、画面方針、権限設計、CSV 設計、基本動作ルールの確定内容を記録する
- 未決事項の一次ソースは [design-decisions-checklist.md](/C:/wk/csherp/PartWire/docs/jp/design-decisions-checklist.md) とする
- 未決事項が確定した場合は、領域単位で本書へ反映する
- 設計判断の根拠や保留事項は本書ではなく決定ログ側で継続管理する

## 1. 目的

本書は、要件定義書をもとに `PartWire` の基本設計方針を整理するための初版である。現時点では、少人数運用に適したシンプルな構成を優先し、後から拡張できる設計を目指す。

## 2. 設計方針

- 案件を中心に、見積、承認、発注、納品、請求、回付を時系列で管理する
- 工事番号と案件番号を分離し、工事番号は最上位集計単位、案件はその配下の業務運用単位として扱う
- 部品マスタと価格履歴を分離し、価格の変遷を蓄積する
- 購入部品と製作部品を同一業務フローで扱えるようにしつつ、属性は別管理とする
- 納品確認を最重要操作とし、一覧性と入力効率を優先する
- CSV 取込を標準機能とし、マスタ整備と既存データ移行に対応する
- 将来的な添付ファイル管理や通知機能の追加を想定し、拡張余地を持たせる
- 見積書は取引単位として扱い、進捗と完了判定は明細レベルで積み上げる

## 3. 想定アプリ構成

現行リポジトリの構成から、まずはデスクトップアプリとして開始する前提で整理する。

- クライアント: WPF
- 永続化: SQLite または SQL Server を想定
- 入出力: 画面操作、CSV 取込、CSV 出力
- 配布: インストーラー

補足:
- 実運用人数が少ないため、初期段階は過度に複雑な分散構成を避ける
- 確認用のローカルスタンドアロン版は SQLite を想定する
- 複数人同時利用を行う場合は SQL Server 系共有 DB を想定する
- 認証方式はローカル認証と AD 認証の切り替えを想定する
- Active Directory 認証では Windows ログオン連携を想定する

- 起票済
- 見積依頼中
- 見積取得中
- 一部承認済
- 全見積承認済

- 全発注済

- 納品確認進行中
- 請求進行中

補足:
- 案件状態は案件明細や後続伝票の進捗をもとに自動判定し、手動補正は行わない
- `見積依頼中` と `見積取得中` は分けず `見積取得中` に統合する
- `納品進行中` と `納品確認進行中` は別状態として扱う
- `完了` は、発注対象の全明細で必要な納品確認が完了し、請求書管理要否が有効な取引先については請求完了し、回付対象書類がすべて回付済みになった時点で遷移する
- 分納中は全数量の納品確認が終わるまで `完了` にしない
- `回付済` は回付対象書類がすべて回付された時点で遷移する
- `中止` は手動却下に加え、発注対象の見積がすべてキャンセルまたは失効した場合、または一部発注後でも残りを進めないと判断した場合に遷移可能とする

### 5.5 承認ルート決定方針

- 案件または見積書に設定された部署をもとに承認ルート候補を抽出する
- 見積金額から金額ランクを判定する
- 部署と金額ランクに対応する承認ルート定義から必要承認段階を決定する
- 承認者候補は独自ユーザー情報または AD グループ参照結果から解決する
- 承認は見積書単位の多段階承認とする
- 差戻し後は同一見積書に対して再申請できるようにする
- 案件個別の承認者差し替えや承認ルート上書きは許容しない
- 金額ランクや承認ルート定義の変更は、変更後に新たに起票または申請される案件へ適用する
- 見積書が改版された場合は、旧版の承認結果を引き継がず再承認を必要とする

### 5.6 見積書改版時の扱い

- 見積書改版は納品前まで許可する
- 見積書改版時は旧版見積書をキャンセル扱いとする
- 旧版に紐づく未発注データは新しい版へ引き継がない
- 旧版に紐づく発注済データがある場合は、旧版の納品予定日を無効化し、発注キャンセルを記録して履歴を保持する
- 旧版に紐づく後続データは履歴として保持しつつ、現行進行対象から外す
- 新版見積書は別途承認を取り直したうえで発注する
- 納品後の変更は見積書改版ではなく、別見積または追加発注で処理する

### 5.7 認証方針

- システム設定で AD 認証またはローカル認証を切り替える
- 認証方式に関わらず、業務上の利用者レコードを共通で利用する
- AD 認証時は AD の UPN を AD 連携キーとして用いて利用者を特定する
- ローカル認証時はログインIDとパスワードハッシュで認証する
- AD グループ参照は認証方式とは独立して有効化可否を設定できるようにする
- 権限判定は通常時、独自ユーザー情報と AD グループ参照を併用して解決する
- AD グループ参照が失敗した場合は、独自ユーザー情報のみで継続運用する

## 6. 主要画面設計案

### 6.1 ダッシュボード

目的:
- 当日対応が必要な案件を把握する

主な表示:
- 未見積回答
- 承認待ち
- 発注未処理
- 納期超過
- 納品確認待ち
- 請求書未登録
- 回付未完了

補足:

- ダッシュボードは、案件のどこかに名前がある利用者向けに対象案件を表示する
- その他の案件は、未承認一覧検索や未終了案件一覧検索などの横断一覧から参照できるようにする

### 6.2 案件一覧

目的:
- 条件検索と進捗確認を行う

主な項目:
- 案件番号
- 工事番号
- 案件名
- 依頼者
- 担当者
- 状態
- 最新更新日

補足:

- 承認や納品確認は案件主軸を補完する特化画面として別導線を持たせる

### 6.3 案件詳細

目的:
- 案件に紐づく全情報を一画面で把握する

主な領域:

- 工事番号

- 承認履歴
- 発注一覧

### 6.4 納品確認画面

目的:
- 最小操作で納品確認を完了する

設計方針:
- 未確認データを優先表示する
- サプライヤー、納品日、案件番号、納品書番号で素早く絞り込めるようにする
- 一覧上で確認数量、差異、確認結果を直接入力できるようにする
- キーボード主体で登録できるようにする
- 利用デバイスは PC を前提とする
- 部分納品や差異ありをその場で記録できるようにする
- 見積書単位の進捗と、明細ごとの残数量を同時に把握できるようにする
- 納品書単位で対象を開き、1 件ずつ確認完了できる導線を優先する
- 案件横断の一括確認は行わない
- バーコードまたは QR コード読取はオプション機能として追加可能なモジュール構成を想定する

### 6.5 納品書登録画面

目的:
- 見積書をもとに納品書を素早く登録する

設計方針:
- 対象見積書を選択して、その明細を初期表示する
- 見積書明細ごとの残納数量を表示する
- 納品数量を明細ごとに入力できるようにする
- 分納済み実績を参照しながら入力できるようにする
- 納品書番号、納品日、添付リンクをあわせて登録できるようにする
- 購入部品ではサプライヤー候補を、製作部品では加工メーカー候補を優先表示する

### 6.6 請求書登録画面

目的:
- 納品実績をもとに請求書を素早く登録する

設計方針:
- 対象納品書または未請求納品明細を選択して初期表示する
- 納品明細ごとの未請求数量、未請求金額を表示する
- 今回請求対象とする数量、金額を入力できるようにする
- 請求書登録対象は、取引先マスタの請求書管理要否が有効な取引先のみとする
- 1 件の請求書に紐づける納品は同一見積書に属するものに限定する
- 見積書完了判定は数量基準とし、見積額との差異がある場合は通知対象として扱う
- 値引き額、雑費、管理費、輸送費などを自由記述で登録できるようにする
- 月またぎ分納や分割請求の履歴を参照しながら入力できるようにする
- 月またぎ分納時も専用画面は設けず、案件に対する請求登録の流れを維持する
- 請求書番号、請求日、支払対象月、添付リンクをあわせて登録できるようにする

### 6.7 マスタ管理画面

対象:

### 6.8 CSV 取込画面

- ただし設定系の重要テーブルは監査対象に含める
- 監査ログは変更前後の主要項目を保持する標準粒度とする
- 保持期間は標準 5 年とし、設定変更可能にする
- 監査ログの閲覧権限はシステム管理者のみに限定する

## 11. 技術設計上の追加方針

- DB 製品差異はデータアクセス層で吸収する
- SQLite から SQL Server への移行手段として、専用移行ツールまたは管理者向けコマンドを用意する
- 移行ツールは添付ファイルリンクを変更せず、そのまま移行対象として扱う
- 主要テーブルには楽観ロック用の更新日時を持たせる
- 更新競合時は競合ありと表示し、画面更新のうえ再入力を促す
- 添付保存先は設定変更可能とし、標準は共有フォルダ、追加対象として SharePoint を含める
- サンプル読み込み機能により初期データ投入を行えるようにする
